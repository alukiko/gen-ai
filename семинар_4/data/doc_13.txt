Vladimir Okladnikov
Нужно сделать Мультиагентную систему для решения бенчмарка GAIA.
Составные части
1) Planer
Что делает:
- получает query,
- разбивает задачу под задачи
- распределяет задачи между агентами
- передает итоговый план Оракулу (последовательность связок агент - подзадача)
Чем обладает:
- system промт
- список всех агентов с их навыками
2) Оракул
Что делает:
- получает итоговый план распределения подзадач между агентами
- Распределяет ресурсы между агентами (выдает LLM и кол-во токенов) получается план в виде связок
(Агент - подзадача - LLM - число токенов)
3) Агенты
Что делают:
- получают подзадачи и ресурсы, которые используют для решения своих задач
Чем обладают:
- system промт - с ролью
- некоторый набор тулов
- ресурсы в виде LLM и токенов для каждой задачи
Какие могут быть роли у агентов
1.Поисковик
2. Итоговый ответчик
3. Критик
4. (Возможно нужны какие-то ещё это пока набросок)
Возможно стоит взять за базу существующую Агентную систему для решения этого бенчмарка и просто модифицировать её оракулом раздающем ресурсы.

Привет!
Вот так новости. Напишу пока сюда размышления насчет теоретических выкладок, которые за это время получилось довести.
1) Полиномиальная вычислимость
Поскольку у нас assignment problem (каждый агент получает ровно один бандл), задача сводится к взвешенному паросочетанию с ограничением на кратность. Оптимальное взвешенное распределение по нашей функции оптимизации = (1−λ)·NSW/NSW* − λ·UE/UE* находится точно за полиномиальное время при фиксированном числе агентов.
Следствие:
наша система легко масштабируется с ростом числа задач и моделей.
2) Нижняя оценка на NSW нашего решения
Если кубик оценок ε-разнообразен (агенты различаются в предпочтениях хотя бы на ε·max_utility), то наше решение σ* гарантирует:
NSW(σ*) ≥ (1 − λ + λ·ε) · NSW*
То есть чем разнообразнее оценки агентов, тем ближе мы к оптимальному NSW, что является мерилом оптимальности нашей системы с точки зрения эффективной раздачи ресурсов.
3) Нижняя оценка на accuracy
Если полезность u_i(m) = p_i(m) + ξ, где p_i(m) — реальная вероятность выполнения задачи, и ξ – δ-калиброванная ошибка оценщика ( |ξ| ≤ δ ), то:
Accuracy(σ*) ≥ (1−λ)
/e
· Accuracy(σ_opt) − 2δ
То есть наша система не хуже идеального назначения в e/(1−λ) раз, плюс аддитивный штраф 2δ за ошибку оценщика.
Влияние на архитектуру нашего рещения
: можем в явном виде задать
практическое правило
выбора регуляризатора
λ* = min(1 − e·(α_min + 2δ) / Accuracy(σ_opt), ε)
Влияние на эксперименты
:
Эксп. 1: можно явно измерить δ — корреляцию предсказанных u_i и реальных success rate. δ входит в гарантию напрямую. Также помимо общей корреляции, явно репортить δ и ε для каждого оценщика, и на основании этих численных результатов сделать подстановку в наши теоретические результаты, чтобы дать гарантированную нижнюю аценку на accuracy по сравнению с идеальным выполнением, ещё до запуска Эксп. 2.
Эксп. 1: можно измерить ε-разнообразие кубиков — дисперсию оценок по агентам для фиксированной модели.
Эксп. 3 (сетка по λ): теорема предсказывает монотонное снижение нижней границы accuracy с ростом λ — можно сравнить с эмпирической кривой.
